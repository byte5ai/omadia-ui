# omadia-canvas-protocol/1.0

> The wire format between Tier 1 (Canvas Host App + `omadia-ui-channel`) and
> Tier 2 (`omadia-ui-orchestrator`). This is the **protocol reference**;
> `CONCEPT.md` holds the design rationale. The machine-validatable truth lives in
> [`schema/`](schema/) — this document explains semantics, lifecycle, and the
> parts a JSON Schema cannot express. Where prose and schema disagree, the schema
> wins.

**Status:** `1.0` baseline. Validated: the schemas compile and a representative
set of accept/reject cases passes under a draft-2020-12 validator (see the
`schema/` notes). Drafted ahead of the spike (Phase 0 of `implementation-plan.md`)
so the Tier-1 validator and the Tier-2 composition share one source of truth.

## 0. Conventions

- All messages are UTF-8 JSON, one JSON value per WebSocket frame.
- **Versioning is two-axis.** The **protocol** (`1.0`, this doc) and the
  **ops-catalog** (`1.0`) are versioned independently and negotiated separately
  at boot — the catalog can grow faster than the wire grammar. A **minor** bump
  (`1.1`) is additive: old clients ignore unknown fields/types/operations. A
  **major** bump (`2.0`) is breaking; reserved, not expected in the v1 lifecycle.
- **Opaque `RevisionId`.** Every `treeRevision`/`producesRevision`/
  `basedOnRevision`/`revision` value is opaque and compared by **equality only**
  — never `<`/`>`/arithmetic. v1 implements it as a monotonic integer rendered as
  a string; v2+ may use Lamport clocks / CRDT op-ids with no wire change.
- **Stable IDs are the lingua franca.** Targets and patches address data by
  stable id/hash (`rowKey`, `itemKey`, `pointKey`, `bufferContentHash`,
  `textRangeAnchor`), never by on-screen position. "Row 3" is not addressable.

## 1. Boot handshake — [`schema/handshake.schema.json`](schema/handshake.schema.json)

Server-initiated, first exchange after WebSocket-open. The server mints a
`handshakeId` in `handshake_offer`; **every** reply (`handshake_select`,
`handshake_error`, `handshake_ack`) echoes it as the correlation key for the
(possibly once-retried) exchange — the schema requires `handshakeId` on all four
messages.

1. **Server → Client `handshake_offer`** — `protocolVersions`, `opsCatalogVersions`, `serverFeatures?`, `handshakeId`.
2. **Client → Server `handshake_select`** — one chosen `protocolVersion` + `opsCatalogVersion`, `clientFeatures?`, and **`localOperations`** (the catalog ops this client actually implements — the authoritative routing truth for Class-B actions).
3. **Mismatch → Server `handshake_error`** — `reason ∈ {protocol-version-unsupported, ops-catalog-version-unsupported, local-ops-incomplete}` + `supported`. The client may downgrade and re-`select` **once**; a second mismatch closes the connection.
4. **Success → Server `handshake_ack`** — the stream phase begins; surface events flow.

`localOperations` is load-bearing: a Host App can ship a subset of the catalog and still work — Tier 2 routes a Class-B op to the client only if the client declared it, else falls back to a Tier-3 tool. Composition idioms degrade gracefully.

## 2. Primitive tree — [`schema/canvas-tree.schema.json`](schema/canvas-tree.schema.json)

**24 primitives** in three groups, rendered by Tier 1 against the single Omadia
(Lume) theme. The root is any primitive; `container`/`pane`/`tabs`/`form`/
`toolbar` nest children recursively.

| Group | Primitives |
|---|---|
| **Core — data + UI** (20) | `text` `heading` `container` `list` `table` `tree` `button` `input` `choice` `toggle` `image` `chart` `form` `toolbar` `menubar` `tabs` `pane` `status` `progress` `divider` |
| **Editor-class** (4) | `media` `canvas-region` `timeline` `vector-path` |

Editor primitives carry required props (`media`: `mediaType`/`dataRef`/`duration`;
`canvas-region`: `width`/`height`/`pixelFormat`; `timeline`: `tracks`/`timebase`;
`vector-path`: `points`) — see the schema for the full set.

**Validation is strict (the whitelist parser).** The schema uses
`unevaluatedProperties: false` per primitive, so an **unknown primitive `type`**
or an **unknown prop/trait** is rejected — exactly the security guarantee
`CONCEPT.md` §Security requires. Tier 1 rejects a non-conforming tree hard
(`surface_error`) rather than rendering arbitrary JSON.

### 2.1 Stable-ID requirements (hard, not optional)

| Carrier | Required field |
|---|---|
| `table` row | `rowKey` |
| `list` / `tree` item | `itemKey` |
| `chart` data point | `pointKey` |
| `canvas-region` / `media` buffer | `bufferContentHash` (content-addressed) |
| `text` segment | `textRangeAnchor` (content-hash + offsets, not raw offsets) |
| container | `id` |

Without these, every beam/mutation/referential-continuity breaks the first time
the data reorders or the client filters. The schema marks `rowKey`/`itemKey`/
`pointKey` **required**.

### 2.2 Cross-cutting traits

Every primitive may carry: `id`, `dataRef`, `selectable` (`none|single|multi` —
mode only; the selected ids live client-side in `viewState.selection`),
`loading`, `error`, `virtualized`, `action`, `style` (theme tokens only,
including the `prose`/`mono` typographic registers), `continuous-input`,
`selection-region`, `realtime-output`, `frame-precise-time`, `suggestedActions`,
`dataClass`. Free-form style is clipped by the Tier-1 normaliser.

### 2.3 Mutability + `suggestedActions`

A data-carrying container declares `dataClass` and per-field `editable` +
client-checkable constraints (`type`/`min`/`max`/`pattern`/`enum`/`maxLength`/
`required`); containers declare `canAddItems`/`canRemoveItems`/`canReorder`.
**Default is read-only** — opt-in to mutable. Tier 2 derives these
deterministically from a tool's `writeCapabilities` (no LLM call;
`deriveMutabilityCapabilities`, shipped in omadia core #170). `suggestedActions`
are pre-supplied per container so the action panel opens instantly with no turn;
each carries `id`/`label`/`effect`/`target`/`validUntilRevision?`/
`validWhileDataRefs?`/`prompt?` and is GC'd client-side when its revision or
DataRefs go stale.

## 3. `TargetRef` — [`schema/target-ref.schema.json`](schema/target-ref.schema.json)

The single canonical way to address a target everywhere (beam,
`_pendingMutation.target`, `suggestedActions.target`, `surface_local_action.target`,
mutation-resolution correlation). **Ten variants**, discriminated by `kind`:
`canvas` · `container` · `element` · `rowField` · `item` · `point` · `textRange`
(via `textRangeAnchor`) · `region` (via `bufferRegion`) · `buffer` · `timeRange`.
Buffer-native coordinates are zoom/pan-independent; a `bufferContentHash`/
`contentHash` mismatch fails the operation (`surface_error`) rather than
silently re-projecting.

`timeRange` carries `{ primitiveId, bufferContentHash, start, end, unit }` with
`unit ∈ {seconds, samples, frames}`. For a single-buffer medium omit
`trackId`/`clipId`; for a multi-track `timeline`, `trackId` is **required** and
`clipId` optional. That last constraint is **resolver-enforced, not
schema-enforced**: a `TargetRef` carries no multi-track discriminator, so the
schema leaves both optional, and Tier 1 fails a track-less `timeRange` against a
multi-track target (`surface_error`).

## 4. `DataRef` — [`schema/data-ref.schema.json`](schema/data-ref.schema.json)

`{ id, signedToken, expiresAt }`. `id` is content-addressed
(`<kind>-<sha256(content)[:16]>`) → automatic dedup and a stable undo/redo chain
(v2+). Pixel/audio/video/vector buffers live in the Tier-1 client; structured
payloads live in the Tier-2 per-canvas cache. `signedToken` is HMAC-signed
(see §9); the server re-validates signature, scope, and expiry on fetch.
Invalidation is `expiresAt`-driven or an explicit `surface_data_ref_invalidated`.

## 5. Surface event grammar — [`schema/surface-events.schema.json`](schema/surface-events.schema.json)

Tier 2 streams these to Tier 1 (folded into the SDK `ChatStreamEvent` union;
classic channels default-ignore them). Every event carries
`{ canvasSessionId, surfaceSeq }`; `surfaceSeq` is server-assigned and monotonic
per `canvasSessionId` (gaps trigger a snapshot request).

| Event | Causal fields | Purpose |
|---|---|---|
| `surface_snapshot` | `producesRevision` | initial render / full replace; starts a revision (carries `protocolVersion` + `opsCatalogVersion`) |
| `surface_patch` | `basedOnRevision` → `producesRevision` | incremental; client rejects on `basedOnRevision` mismatch and requests a snapshot |
| `surface_data_ref_created` | `revision` | bulk data now available behind a signed `DataRef` |
| `surface_data_ref_invalidated` | `revision` | `{ id, reason }` — reference expired / replaced |
| `surface_action_result` | `forActionId`, `basedOnRevision` | result of a user-triggered action |
| `surface_local_action` | `revision`, `effect` | instruct Tier 1 to run a catalog op. `effect: preview` does **not** mutate the revision; `effect: durable` is **always** followed by a `surface_patch` (shared-canvas-safe by construction) |
| `surface_error` | `revision` | render-side validation / dataRef denied / unknown op / protocol mismatch |
| `surface_mutation_resolved` | `revision`, `forMutationId` | resolves a `_pendingMutation` with `status ∈ {success,modified,rejected,invalid,conflict}`; reserves `originAuthor`/`originSession` for v2 |

**Client rules.** Snapshots reset to `producesRevision`. Patches require an
equality match on `basedOnRevision`, else drop + request a snapshot.
`surface_local_action` runs against current local state (no revision change
unless a patch follows). `surfaceSeq` is the transport tie-breaker.

### 5.1 Patch op grammar (pinned in the spike)

`surface_patch.patches` items are an RFC-6902 subset:
`{ "op": "add" | "replace" | "remove", "path": <JSON-Pointer>, "value"? }`.
`add` supports the `-` append index. Positional array segments are valid
*inside* a patch because `basedOnRevision` equality pins the tree shape the
path was authored against; the stable-ID rule governs cross-turn references,
not intra-patch addressing. Reference implementation:
`app/src/shared/treePatch.ts` (Host App) — Tier 2 must emit only this subset.

**Snapshot re-request.** v1 defines no dedicated wire message: on a
`surfaceSeq` gap or `basedOnRevision` mismatch the client closes the socket
and reconnects, re-`handshake_select`ing with its persisted
`canvasSessionId`; the server answers the next turn (or, once Tier-2 canvas
state persistence lands, the re-handshake itself) with a fresh
`surface_snapshot`.

**Open feedback (v1.1 candidates):** (a) `tableRow` is closed
(`additionalProperties: false`), so a row-highlight patch (Walkthrough 1
step 13) has no schema-valid wire shape yet; (b) ~~the client `turn` message
carries `action` (button/row clicks) which rides
`IncomingTurn.metadata.action` until the SDK grows a typed field~~ —
**landed**: `ChatTurnInput.action` (`{ type, payload?, target? }`) is the
typed field; the dispatcher threads `metadata.action` + the turn's `target`
into it and the canvas orchestrator hands it to the main turn as a
`[canvas-action]` block (omadia PR #277).

## 6. Upstream turn — `IncomingTurn` + `viewState`

§5 is the downstream stream (Tier 2 → Tier 1). The **upstream** message is the
turn the channel forms when the user does something that needs the agent (beam,
form submit, external action) and hands to the orchestrator as an `IncomingTurn`
(SDK type, omadia core #167). Canvas-relevant additive fields: `tenantId`,
`target` (a `TargetRef`, §3), `viewState`, `viewStateTruncated`.

`viewState` is the per-container client view-state snapshot
(sort/filter/group/pagination/**selection**) carried *alongside* the turn for
referential continuity. **Tier 2 reads it, never writes it** — mutating it would
violate the authority split. Selection is not persisted canvas state; it lives
only in `viewState.selection` as stable IDs (§2.1), delivered with the turn.

**Truncation is a wire contract; the budget that triggers it is sender policy.**
The protocol normalises only the parts Tier 2 must interpret:

- **Truncation envelope.** When a `selection` is too large to ship in full, it is
  replaced by `selection: { kind: "truncated", includedCount, totalCount, sample }`
  — a clearly-marked prefix `sample` in viewState ordering. Tier 2 may fetch the
  full selection via a Tier-3 lookup if it needs it.
- **Flag + normative rule.** A truncated turn carries `viewStateTruncated: true`
  (the field #167 landed on `IncomingTurn`). On that flag the Tier-2 skill **MUST
  ask before mutating across the truncated selection** (explicit user
  confirmation).

What stays **out** of this versioned spec, because it is sender-side policy that
must be re-tunable without a protocol bump: the thresholds that make Tier 1
truncate (per-turn `viewState` size cap, per-selection ID cap) and the
changed-containers-only / reverse-priority eviction strategy. These are
spike-tunable and live in `CONCEPT.md` (§ `viewState` payload budget) as initial
defaults.

`IncomingTurn` itself is defined by the SDK types (#167); the truncation envelope
is the one viewState-specific wire shape, normalised here in prose. No separate
`view-state` schema in v1 — the SDK type binds the shape, and the budget is
policy, not wire norm.

## 7. Sentinel envelopes — [`schema/sentinels.schema.json`](schema/sentinels.schema.json)

Tier-3 tools (and, for `_pendingMutation`, the client) hand structured payloads
to the orchestrator as in-band JSON sentinels in the result string; the
orchestrator `JSON.parse`-extracts them. **Origin-gated:** the canvas sentinels
(`_pendingCanvasTree`, `_pendingStructuredPayload`) are accepted only from a tool
whose plugin declares the `canvas-output` capability (parsers + gate shipped in
omadia core #169; the boot allow-set + wiring land with the orchestrator
consumer). `_pendingMutation` is the client→orchestrator Class-D optimistic
update (`mutationId`/`target`/`oldValue`/`newValue`/`basedOnRevision`). Classic
sentinels (`_pendingUserChoice`, `_pendingRoutineList`) are unchanged.

## 8. Local operations catalog (Tier 1)

Declared in `handshake_select.localOperations`. Each op has an `effect`:
**`preview`** (transient, undo-able locally, no revision change) or **`durable`**
(mutates the buffer/structure; Tier 2 follows with a revisioned `surface_patch`).
v1.0 baseline:

| Domain | Operations | Effect |
|---|---|---|
| Pixel | brush · erase · fill · blur · sharpen · levels · curves | durable |
| Pixel transforms | crop · resize · rotate · flip | durable |
| Pixel preview | preview-blur · preview-curves · preview-levels | preview |
| Vector | move · scale · rotate · smooth · bezier-edit | durable |
| Audio | trim · fade · normalize · gain · mute | durable |
| Audio preview | preview-gain · preview-eq · scrub | preview |
| Video | trim · splice · speed · mute-track | durable |
| Video preview | scrub · preview-speed | preview |
| Geometry | move · rotate · scale · snap | durable |
| Layer | visibility · opacity · blend-mode · lock · reorder | durable |
| Selection | rectangle · lasso · magic-wand · invert · deselect | preview (mutates `viewState.selection`, not the tree) |

Catalog versioning aligns with the protocol; new operations require a minor bump
and are negotiated separately at boot.

## 9. Security surface

- **`DataRef` HMAC** — `signedToken = HMAC(serverSecret, tenantId ‖ userId ‖ canvasSessionId ‖ dataRefBody ‖ expiryEpoch)`. Rotating secret (24h), `expiresAt ≤` secret lifetime, graceful rotation (next-secret accepted alongside current for one period).
- **Whitelist parser** — unknown primitive `type` or trait → rejected (§2, enforced by the schema).
- **Closed ops catalog** — only catalog-listed operations execute; unknown op → `surface_error`.
- **Action-effect classification** — each `action` carries `effect ∈ {local, internal, external-effect}`. An `external-effect` action MUST be gated behind a confirmation modal; its tool fires only after the user emits the `confirm-<actionType>` action — the original action never invokes its tool directly.
- **Sentinel origin-gating** — canvas sentinels rejected unless the producing tool declares `canvas-output` (§7).

## 10. Versioned components

Primitive vocabulary · cross-cutting traits · surface event grammar · local-ops
catalog baseline · sentinel envelope · handshake. A 21st primitive or a new op is
a minor bump via RFC + PR, documented under `docs/protocol/<version>.md`.

## 11. Notifications (v1.1 additive — omadia-ui#15)

Notifications are **out-of-band from the canvas surface**: the canvas tree is
the *data* channel; user notifications ride their own message on the same
authenticated WebSocket. Producer side is the middleware `NotificationRouter`
(the `ui-channel` registers a channel handler and fans out to the target
user's live sockets).

**Wire shape.** Server → client `notification`:
`{ id, severity: info|success|warning|error, title, body?, source?,
action?: { type, payload?, label? }, dedupeKey?, ttlMs?, scope? }` —
structured and server-authoritative, never free text only. Client → server
`notification_ack { id }` on seen/dismiss (server-side history sync is a
later slice).

**Severity → UI element (fixed mapping).**

| Severity | Element | Behaviour |
|---|---|---|
| `info` / `success` | Toast (bottom-right stack) | auto-dismiss (`ttlMs`, default 6 s), non-blocking |
| `warning` / `error` | Banner (top, centered) | persists until dismissed; carries the typed `action` |
| any | Bell history | persisted client-side (max 100), unread badge |
| turn-scoped errors | Beam chip (existing) | unchanged — not part of this channel |

**Do:** structured payloads only; dedupe/coalesce via `dedupeKey`; typed
`action` reuses the canvas action→turn plumbing; `role="status"` for info,
`role="alert"` for errors; run text through the channel's privacy guard.

**Don't:** never render notifications into the canvas tree; no blocking
modals for non-critical info; no per-tool-call spam (the canvas already shows
progress); never auto-dismiss errors or actionable items; no secrets/PII/raw
stack traces; the debug `notices` buffer stays an internal diagnostic strip.
